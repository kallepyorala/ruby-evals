# Cross-Language Eval Framework Survey

*Primary-source survey of the leading AI-eval frameworks (Python/TS/YAML) — extracting conceptual models, vocabulary, and mechanisms to inform a new Ruby-first library for evaluating AI apps and agents.*

**Date:** 2026-07-17

Frameworks studied from primary sources: **Pydantic Evals**, **Braintrust `Eval()` + autoevals**, **promptfoo**, **DeepEval**, **Ragas**, **OpenAI Evals**, and **Inspect** (UK AISI, skim). Every nontrivial claim is cited to an official doc page or a source file; version/date noted where visible. A few points are flagged as unverified where a primary source could not be reached.

---

## Executive summary

- **The shared vocabulary is remarkably stable.** Almost every framework converges on: a **Case / Sample / test** (one input + optional expected output + metadata), collected into a **Dataset**, run against a **Task** (the callable under test), scored by one or more **Evaluators / Scorers / Assertions / Metrics**, producing a **Report / Experiment**. The load-bearing disagreement is what to call the scorer: Pydantic says **Evaluator** ([source](https://pydantic.dev/docs/ai/evals/evals/)), Braintrust/autoevals say **Scorer** returning a **`Score`** ([SCORERS.md](https://raw.githubusercontent.com/braintrustdata/autoevals/main/SCORERS.md)), promptfoo says **assertion** ([guide](https://www.promptfoo.dev/docs/configuration/guide/)), DeepEval/Ragas say **Metric** ([DeepEval](https://deepeval.com/docs/metrics-introduction), [Ragas](https://github.com/explodinggradients/ragas)), Inspect says **Scorer** ([scorers](https://inspect.aisi.org.uk/scorers.html)).
- **The sharpest design fork is score semantics.** Two camps: (a) **boolean-per-case → accuracy** (OpenAI Evals `Match`/`Includes` → `metrics:[accuracy]` ([templates](https://github.com/openai/evals/blob/main/docs/eval-templates.md))); (b) **numeric [0,1] score + a threshold/pass condition** (Braintrust `score∈[0,1]|null` ([README](https://raw.githubusercontent.com/braintrustdata/autoevals/main/README.md)), DeepEval `score≥threshold` ([metrics](https://deepeval.com/docs/metrics-introduction)), promptfoo weighted-average vs `threshold` ([expected-outputs](https://www.promptfoo.dev/docs/configuration/expected-outputs/))). **Pydantic Evals unifies both**: the evaluator's Python return *type* decides — `bool`→assertion, `int/float`→score, `str`→label, `Mapping`→several named results ([evaluator.py](https://raw.githubusercontent.com/pydantic/pydantic-ai/main/pydantic_evals/pydantic_evals/evaluators/evaluator.py)). Ragas is the outlier: **bare floats, no threshold, mean-only aggregation** — pass/fail is the caller's problem ([base.py](https://github.com/explodinggradients/ragas)).
- **LLM-judge design has collapsed to one recipe.** A judge is a **prompt/rubric template + a fixed set of choices + a choice→number map + optional chain-of-thought**, parsed via structured output. Braintrust `LLMClassifier(prompt_template, choice_scores, use_cot)` ([README](https://raw.githubusercontent.com/braintrustdata/autoevals/main/README.md)), OpenAI modelgraded YAML (`prompt`/`choice_strings`/`choice_scores`/`eval_type: cot_classify`) ([templates](https://github.com/openai/evals/blob/main/docs/eval-templates.md)), and Pydantic `LLMJudge`→`GradingOutput(reason, pass_, score)` ([llm_as_a_judge.py](https://raw.githubusercontent.com/pydantic/pydantic-ai/main/pydantic_evals/pydantic_evals/evaluators/llm_as_a_judge.py)) are the same shape. The `Factuality` graded rubric (A–E → `0.4/0.6/1/0/1`) literally recurs across Braintrust, OpenAI, and DeepEval lineage.
- **Two distinct "custom judge" archetypes worth stealing.** Free-form **G-Eval** (criteria → auto-generated CoT steps → 1–10 score, token-probability-normalized) exists in DeepEval ([metrics-llm-evals](https://deepeval.com/docs/metrics-llm-evals)) and Pydantic (`GEval`); and **decision-tree judging** — DeepEval's `DAGMetric` ([metrics-dag](https://deepeval.com/docs/metrics-dag)) — that gates a subjective LLM check behind deterministic branches for reproducibility. Ragas contributes the **decompose→per-unit verdict→aggregate** idiom (Faithfulness: split answer into atomic statements, NLI each, score = fraction supported).
- **Statistical rigor is the biggest capability gap between the leaders and everyone else.** **Inspect** is alone in treating it as first-class: `stderr` on every metric, and **epochs + reducers** (`mean`/`median`/`mode`/`max`/`at_least(k)`/`pass_at(k)`) for repeated trials ([scorers](https://inspect.aisi.org.uk/scorers.html), [reducers](https://inspect.aisi.org.uk/reference/inspect_ai.scorer.html)). Everyone else offers a bare repeat count (`trial_count` / `--repeat` / `repeat` / `-r`) that re-runs but reports no variance; Pydantic adds **report-level statistical evaluators** (confusion matrix, PR/ROC AUC, KS) that run once over all cases ([report-evaluators](https://pydantic.dev/docs/ai/evals/evaluators/report-evaluators/)); Ragas offers only `strictness` majority-voting ([_aspect_critic.py](https://github.com/explodinggradients/ragas)).
- **Agent/tool-call eval has matured fast and is now table stakes.** Pydantic parses an **OTel span tree** into `ToolCorrectness`/`TrajectoryMatch`/`ArgumentCorrectness`/`MaxToolCalls`/`MaxModelRequests` ([agentic.py](https://raw.githubusercontent.com/pydantic/pydantic-ai/main/pydantic_evals/pydantic_evals/evaluators/agentic.py)); DeepEval turns `@observe` spans into scored test cases plus `ToolCorrectnessMetric`/`TaskCompletionMetric` ([component-level](https://deepeval.com/docs/evaluation-component-level-llm-evals)); promptfoo has `trajectory:*` assertions and a built-in OTLP receiver ([tracing](https://www.promptfoo.dev/docs/tracing/)); Ragas has `ToolCallAccuracy`/`AgentGoalAccuracy`/`TopicAdherence` over multi-turn message lists. Notably **`ToolCorrectness`/`ToolCallAccuracy` are deterministic**, not LLM-based, in both DeepEval and Ragas.
- **Config-as-data vs code-first is a real philosophical split.** promptfoo is **YAML-config-first** (`promptfooconfig.yaml`, prompts × providers × tests matrix) ([guide](https://www.promptfoo.dev/docs/configuration/guide/)) and OpenAI Evals is **registry-YAML + JSONL**; Pydantic, Braintrust, DeepEval, Ragas are **code-first** but every one of them also serializes datasets to/from YAML/JSON/CSV/JSONL. Pydantic goes furthest, auto-generating a companion **JSON Schema** for editor autocomplete on YAML datasets ([serialization](https://pydantic.dev/docs/ai/evals/how-to/dataset-serialization/)).
- **The three ideas most worth borrowing for Ruby:** (1) **a single uniform scorer contract** — a plain callable returning a small value object `{name, score, pass?, reason}`, with deterministic and LLM judges as the *same* interface (Braintrust's `Score`/promptfoo's `{pass,score,reason}`); (2) **the thin LLM-judge layer** — rubric + choices + choice-scores + CoT over that same contract, judge model pluggable at 3 precedence levels (promptfoo's CLI < defaultTest < per-assertion); (3) **epochs/reducers with real variance reporting** (Inspect) — Ruby has no leader here and it is the clearest differentiation opportunity. The clearest thing to **reject** is Ragas' no-threshold float-only model and OpenAI Evals' YAML-template-fused eval logic.

## 1. Pydantic Evals

Primary sources: [pydantic.dev/docs/ai/evals/](https://pydantic.dev/docs/ai/evals/evals/) (formerly `ai.pydantic.dev/evals`, now 301-redirects), and `pydantic_evals/` source in the `pydantic-ai` monorepo. **Version 2.9.0/2.9.1 (2026-07-11)** ([PyPI](https://pypi.org/project/pydantic-evals/)). Deps: `pydantic-ai-slim`, `rich>=13.9.4`, `logfire-api>=3.14.1`. The current `main` is materially richer than older write-ups (adds agentic + report-level statistical evaluators).

**Conceptual model.** `Case` = "a single test scenario … with optional expected outputs, metadata, and case-specific evaluators"; `Dataset` = a collection of Cases; `Evaluator` = the scorer base class; `Task` = the `Callable[[InputsT], OutputT]` under test; `Experiment` = the conceptual name for one run ("one Experiment evaluates one defined Task"), yielding an `EvaluationReport`. It brands itself **code-first**. It uses **Evaluator**, not "scorer"; "judge" appears only as `LLMJudge`. ([evals overview](https://pydantic.dev/docs/ai/evals/evals/))

**Dataset & case representation.** `Case(*, name=None, inputs, metadata=None, expected_output=None, evaluators=())` — all keyword-only; everything but `inputs` optional. `Dataset(*, name, cases, evaluators=(), report_evaluators=())`. Datasets serialize with `to_file(path, schema_path='{stem}_schema.json', …)` / `from_file(...)` (YAML or JSON by extension) and **auto-generate a companion JSON-Schema file** so editors give autocomplete + inline validation (`# yaml-language-server: $schema=…` for YAML, `"$schema"` key for JSON). Evaluators serialize compactly by name (`EqualsExpected`), single positional arg (`IsInstance: str`), or dict form. Custom evaluators must be `@dataclass`, subclass `Evaluator`, and be passed via `custom_evaluator_types=[…]` to both ends. ([dataset.py](https://raw.githubusercontent.com/pydantic/pydantic-ai/main/pydantic_evals/pydantic_evals/dataset.py), [serialization](https://pydantic.dev/docs/ai/evals/how-to/dataset-serialization/))

**Evaluator interface & score semantics (the signature idea).** `Evaluator.evaluate(ctx: EvaluatorContext) -> EvaluatorOutput` (sync or async). The return type disambiguates result kind:
```
EvaluationScalar = bool | int | float | str
EvaluatorOutput  = EvaluationScalar | EvaluationReason | Mapping[str, EvaluationScalar | EvaluationReason]
```
Rule, verbatim: **"`int` and finite `float` are treated as scores, `str` as labels, and `bool` as assertions"**; a `Mapping` yields multiple named results from one evaluator. `EvaluationReason{value, reason}` attaches a justification. `EvaluatorContext` bundles `inputs`, `output`, `expected_output`, `metadata`, `duration`, `metrics: dict`, `attributes`, and `span_tree`. Built-ins: `Equals`, `EqualsExpected`, `Contains`, `IsInstance`, `MaxDuration`, `LLMJudge`, `GEval`, `HasMatchingSpan`. (A legacy `Python` arbitrary-expression evaluator was **removed**.) Evaluators attach at dataset, case, or via `add_evaluator`. ([evaluator.py](https://raw.githubusercontent.com/pydantic/pydantic-ai/main/pydantic_evals/pydantic_evals/evaluators/evaluator.py), [common.py](https://raw.githubusercontent.com/pydantic/pydantic-ai/main/pydantic_evals/pydantic_evals/evaluators/common.py))

**Trials & statistics.** `Dataset.evaluate(..., repeat: int = 1)` exists in source, but its cross-run aggregation semantics are **not documented in prose** (flagged). Reporting aggregates via `ReportCaseAggregate` (score averages, label frequency, assertion pass-rate `n_passing/n_assertions`, duration averages). The real statistical layer is **report evaluators** — `ReportEvaluator` runs once over the full `EvaluationReport` after all cases: built-ins `ConfusionMatrixEvaluator`, `PrecisionRecallEvaluator`, `ROCAUCEvaluator`, `KolmogorovSmirnovEvaluator`. No variance/CI primitive beyond these. ([report-evaluators](https://pydantic.dev/docs/ai/evals/evaluators/report-evaluators/), [reporting/__init__.py](https://raw.githubusercontent.com/pydantic/pydantic-ai/main/pydantic_evals/pydantic_evals/reporting/__init__.py))

**LLM-as-judge.** `LLMJudge{rubric, model, include_input, include_expected_output, model_settings, score, assertion}` — one judge can emit an assertion, a score, or both. Internally the judge is itself a pydantic-ai Agent with `output_type=GradingOutput{reason, pass_, score}` (structured output → schema-enforced). Rubric semantics: "if the statement in the rubric is true … the output passes." Prompt builder assembles `<Input>`/`<Output>`/`<ExpectedOutput>`/`<Rubric>` sections. Default judge model `openai:gpt-5.2`, override via `set_default_judge_model()`. `GEval` is a second judge style (criteria + `evaluation_steps` + `score_range`). ([llm_as_a_judge.py](https://raw.githubusercontent.com/pydantic/pydantic-ai/main/pydantic_evals/pydantic_evals/evaluators/llm_as_a_judge.py))

**Agent / tool-call eval (first-class).** `EvaluatorContext.span_tree` is an OpenTelemetry-derived `SpanTree` of `SpanNode`s; `SpanQuery` is a serializable AND-combined filter (name/attributes/status/timing/children/descendants/ancestors). Purpose-built agentic evaluators parse it: `ToolCorrectness(expected_tools, allow_extra, include_failed)`, `TrajectoryMatch(expected_trajectory, order='in_order')` (exact/LCS/multiset → float 0–1), `ArgumentCorrectness(tool_name, expected_arguments, match_mode='subset')`, `MaxToolCalls`, `MaxModelRequests`. `HasMatchingSpan(query)` turns any span query into a pass/fail. ([agentic.py](https://raw.githubusercontent.com/pydantic/pydantic-ai/main/pydantic_evals/pydantic_evals/evaluators/agentic.py), [span_tree.py](https://raw.githubusercontent.com/pydantic/pydantic-ai/main/pydantic_evals/pydantic_evals/otel/span_tree.py))

**Caching / reproducibility / CI.** `max_concurrency` (unlimited default), `retry_task`/`retry_evaluators` with Tenacity-style `RetryConfig`; exhausted retries yield `EvaluatorFailure`/`ReportCaseFailure`. **No caching mechanism** found (stated as a negative finding). No dedicated CI-runner construct — the pattern is calling `evaluate_sync` from test code.

**Report / terminal UX.** `EvaluationReport{name, cases, failures, analyses, experiment_metadata}`. `report.print()` (Rich), `render(baseline=…)` for before/after diff, `console_table()`/`failures_table()`. Default "Evaluation Summary" table shows Case ID, Inputs, Outputs, per-evaluator Scores, Labels, Assertions (pass %), Durations + an Averages row. **Rich-terminal only — no Markdown/JUnit export**; web viewing is via Logfire. ([reporting/__init__.py](https://raw.githubusercontent.com/pydantic/pydantic-ai/main/pydantic_evals/pydantic_evals/reporting/__init__.py))

## 2. Braintrust `Eval()` + autoevals

Primary sources: [braintrust.dev/docs](https://www.braintrust.dev/docs/platform/experiments), [autoevals README](https://raw.githubusercontent.com/braintrustdata/autoevals/main/README.md), [SCORERS.md](https://raw.githubusercontent.com/braintrustdata/autoevals/main/SCORERS.md), templates (fetched 2026-07-17). **Braintrust already ships a first-party Ruby SDK** — `trial_count` is documented for "Python/Ruby" ([run](https://www.braintrust.dev/docs/platform/experiments/run)); worth diffing a Ruby design against `braintrustdata/braintrust-sdk` directly.

**Conceptual model.** `Eval(name, {data, task, scores, …})`: `data` returns inputs/expected/metadata; `task` maps input→output; `scores` is a list of scorers returning a `Score`. Hierarchy: **Project** (1st arg) → **Experiment** (one immutable snapshot per `Eval()` run) → **Dataset** (reusable cases) with **task**, **data**, **Scorer/`Score`**, **Trial**, **span**, and **Hill Climbing** (reuse a prior experiment's outputs as the baseline `expected`). ([experiments](https://www.braintrust.dev/docs/platform/experiments), [Eval reference](https://www.braintrust.dev/docs/reference/libs/nodejs/functions/Eval))

**Dataset & case representation.** A case (`EvalCase`) is `{input, expected?, metadata?, trial_count?}`. `data` is either an inline function returning dicts or a managed `init_dataset(project, name)`. `expected` is optional (Humor/Security/Moderation need none). ([write](https://www.braintrust.dev/docs/platform/experiments/write))

**Scorer interface & `Score` semantics.** A scorer returns `Score{name, score, metadata}` where **`score` is "a number between 0 and 1, or `None`/`null` when the evaluation is skipped"** — confirming the [0,1]-float-with-null convention (not boolean). A custom heuristic scorer is just a function `(output, expected, input) -> Score`. Scorers instantiate then call (`Factuality()(output, expected, input=…)`; `.eval_async`). Composition = list; each produces its own named column. JSON/list scorers compose sub-scorers (`JSONDiff(string_scorer=Levenshtein, number_scorer=NumericDiff)`). The catalog is explicitly bucketed **LLM-as-a-judge / Heuristic / Statistical**: LLM (`Factuality`, `Battle`, `ClosedQA`, `Humor`, `Security`, `Sql`, `Summary`, `Translation`), RAG (`ContextRelevancy`, `Faithfulness`, `AnswerRelevancy`, …), embedding (`EmbeddingSimilarity`, cosine), heuristic (`Levenshtein`, `ExactMatch`, `NumericDiff`), JSON (`JSONDiff`, `ValidJSON`), list (`ListContains`). ([README](https://raw.githubusercontent.com/braintrustdata/autoevals/main/README.md), [SCORERS.md](https://raw.githubusercontent.com/braintrustdata/autoevals/main/SCORERS.md))

**Trials & statistics.** `trial_count`/`trialCount` runs each input N times "to measure variance"; **absent, each input runs once**. Braintrust "buckets test cases with the same `input` value and computes summary statistics." Experiments are immutable, comparable snapshots; the return object reports "average scores, duration, improvements, regressions." A dedicated "Compare trials" doc exists. ([run](https://www.braintrust.dev/docs/platform/experiments/run))

**LLM-as-judge.** `LLMClassifier(name, prompt_template, choice_scores, use_cot=True)` — the judge selects one of a fixed choice set; `choice_scores` maps each choice string → a number in [0,1]; `use_cot` prepends chain-of-thought. Judge model config has 3 tiers: env vars → global `init(client, default_model)` → per-instance `Factuality(model=…)`; default model `gpt-5-mini`. YAML templates (`templates/*.yaml`) each carry a mustache `prompt` + `choice_scores`. Verified verbatim: **`factuality.yaml`** = 5 choices A–E, `choice_scores: A:0.4, B:0.6, C:1, D:0, E:1` (the canonical graded rubric); `battle.yaml` (Yes:1/No:0), `closed_q_a.yaml` (Y:1/N:0). ([README](https://raw.githubusercontent.com/braintrustdata/autoevals/main/README.md), [factuality.yaml](https://raw.githubusercontent.com/braintrustdata/autoevals/main/templates/factuality.yaml))

**Agent / tool eval.** Tasks accept a `{span}` hook; `traced()` + `span.log({input, output, metrics})` capture multi-step execution as nested spans; agents point to **remote evals** (`/evaluate/remote-evals`) — exact API not fetched (**unverified**). ([run-evaluations](https://www.braintrust.dev/docs/evaluate/run-evaluations))

**Caching / reproducibility / CI.** CLI `bt eval <file|dir>` (Python `eval_*.py`, TS `*.eval.ts`); `--watch` reruns on change; `--no-send-logs`/`no_send_logs` runs locally without creating an experiment; `--first N`/`--json` for smoke tests; auto-loads `.env*`. Caching is provided by the **Braintrust AI Gateway** (automatic request caching for model calls), not a per-example eval cache. `bt eval` returns **non-zero exit on exceptions**; GH Action posts improvement/regression PR comments. ([run](https://www.braintrust.dev/docs/platform/experiments/run))

**Report / terminal UX.** `bt eval` prints a terminal summary (avg scores per scorer, duration, improvements/regressions) + a link to the web experiment; `Eval()` returns `EvalResultWithSummary`. Web UI has side-by-side comparison tables and per-row trace inspection. The exact ASCII terminal-table layout was **not** found on a primary page (unverified).

## 3. promptfoo

Primary sources: [promptfoo.dev/docs](https://www.promptfoo.dev/docs/configuration/guide/) (config reference shows "last updated July 17, 2026").

**Conceptual model.** YAML-config-first: one `promptfooconfig.yaml`, run by `promptfoo eval`. The mental model is a **matrix** — every **prompt** × **provider** (aka **`targets`**) × **test case** is executed and graded. Vocabulary: **prompts**, **providers/targets**, **tests/test case**, **assert/assertion**, **`vars`**, **`metric`** (a tag that aggregates assertions into a named metric), **`threshold`**, **`scenarios`**, **`defaultTest`** (shared defaults, override or `disableDefaultAsserts`), **`derivedMetrics`**, **`redteam`**. ([guide](https://www.promptfoo.dev/docs/configuration/guide/), [reference](https://www.promptfoo.dev/docs/configuration/reference/))

**Dataset & case representation.** `tests` inline or `file://` (CSV, JSON, JSONL, YAML, xlsx, Google Sheets, HF). CSV columns auto-map to `{{vars}}`; reserved columns `__expected`/`__expected1..` (a bare value defaults to `equals`; typed forms `contains:`, `llm-rubric:`, `factuality:`, `similar:`), `__description`, `__metadata:category`, `__metadata:tags[]`. Per-test fields: `description`, `metadata`, `threshold`, `provider(s)`, `providerOutput` (skips API calls), `assertScoringFunction`. Scenarios wrap a `config:` (vars sets) × `tests:` to avoid manual combinatorics. ([test-cases](https://www.promptfoo.dev/docs/configuration/test-cases/))

([tracing](https://www.promptfoo.dev/docs/tracing/), [chat](https://www.promptfoo.dev/docs/configuration/chat/))

## 4. DeepEval

Primary sources: [deepeval.com/docs](https://deepeval.com/docs/evaluation-introduction), [github.com/confident-ai/deepeval](https://github.com/confident-ai/deepeval). **Version 4.1.0 (2026-07-12).** Self-described as "Pytest but specialized for unit testing LLM applications." (Vocabulary note: current v4.x uses `SingleTurnParams` where older docs said `LLMTestCaseParams`.)

**Conceptual model.** `LLMTestCase` (single-turn), `ConversationalTestCase` (multi-turn), `MLLMTestCase` (multimodal). **`Golden`** = "a precursor to a test case … a pending test case" with input/expected but **no `actual_output`** (generated at eval time), reusable across model versions. `EvaluationDataset` = a collection of goldens. `metric`/`BaseMetric`, `assert_test` (raises on failure — for CI), `evaluate()` (returns without raising — for scripts), `deepeval test run` (pytest wrapper), `@observe` (tracing). Two named modes: **end-to-end** and **component-level**. ([evaluation-introduction](https://deepeval.com/docs/evaluation-introduction), [test-cases](https://deepeval.com/docs/evaluation-test-cases))

**Dataset & case representation.** `LLMTestCase(input, actual_output, expected_output?, context?, retrieval_context?, tools_called?, expected_tools?, token_cost?, completion_time?)` — with a deliberate split between `context` (curated ground truth) and `retrieval_context` (what the retriever actually returned); `ToolCall{name, description?, reasoning?, output?, input_parameters?}`. `Golden`/`ConversationalGolden`. `EvaluationDataset`: `add_golden`/`add_test_case`, cloud `push`/`pull`, local `save_as`/`add_goldens_from_{json,csv,jsonl}_file`, `Synthesizer` for synthetic goldens. `ConversationalTestCase{turns:[Turn], scenario, expected_outcome, chatbot_role, …}`. ([datasets](https://deepeval.com/docs/evaluation-datasets), [multiturn](https://deepeval.com/docs/evaluation-multiturn-test-cases))

**Metric interface & score semantics.** `BaseMetric`: `measure(test_case)` / `a_measure`, `.score` (float 0–1), `.threshold` (default 0.5, passes when **score ≥ threshold**), `is_successful()`, `.reason`. Common params: `threshold`, `model` (str or `DeepEvalBaseLLM`), `include_reason=True`, `strict_mode=False` (forces binary 0/1), `async_mode=True`, `verbose_mode`, `evaluation_template`. Compose by passing a list of metrics per case. Catalog spans RAG (`AnswerRelevancyMetric`, `FaithfulnessMetric`, `ContextualPrecision/Recall/RelevancyMetric`), agentic (`TaskCompletionMetric`, `ToolCorrectnessMetric`, `ArgumentCorrectnessMetric`, …), multi-turn (`ConversationalGEval`, `RoleAdherenceMetric`, …), safety (`BiasMetric`, `ToxicityMetric`, `PIILeakageMetric`, …), and custom (`GEval`, `DAGMetric`, `HallucinationMetric`, `JsonCorrectnessMetric`). Scores use LLM statement/claim extraction + verdict classification (e.g. AnswerRelevancy = relevant statements / total). ([metrics-introduction](https://deepeval.com/docs/metrics-introduction))

**Trials / caching / async.** `-r N` repeats; `CacheConfig(use_cache=False, write_cache=True)` (`-c`); `AsyncConfig(run_async=True, max_concurrent=20)`; `a_measure` for true non-blocking. No aggregate variance surfaced beyond re-running (flagged). ([flags](https://deepeval.com/docs/evaluation-flags-and-configs))

**LLM-as-judge (two archetypes).** **`GEval`** — `name, criteria, evaluation_params:[SingleTurnParams], evaluation_steps?` (mutually exclusive with criteria), `rubric?`; algorithm: auto-generate CoT steps from criteria → LLM produces a **1–10** score → **token-probability normalization** to 0–1 (weighted summation of output-token probabilities). `Rubric(score_range, expected_outcome)` constrains bands. **`DAGMetric`** — a `DeepAcyclicGraph(root_nodes)` decision tree (`TaskNode`, `BinaryJudgementNode`, `NonBinaryJudgementNode`, `VerdictNode` with `score` or `child`/`BaseMetric`) for "the same score every run"; gates a subjective `GEval` behind deterministic branches. Judge model = a string or `DeepEvalBaseLLM` subclass; global default via `deepeval set-ollama` etc. ([metrics-llm-evals](https://deepeval.com/docs/metrics-llm-evals), [metrics-dag](https://deepeval.com/docs/metrics-dag))

**Agent / tool eval.** `ToolCorrectnessMetric` — **deterministic, non-LLM** match of `tools_called` vs `expected_tools` (options `should_consider_ordering`, `should_exact_match`, `ToolCallParams.INPUT_PARAMETERS`/`OUTPUT`). `TaskCompletionMetric` — LLM over the **full agent trace** (requires `@observe`). `@observe(metrics=[…])` turns each decorated span into a scored test case (many cases per app run); `update_current_span`/`update_current_trace`. Multi-turn: `ConversationalGEval`, `RoleAdherenceMetric`, `KnowledgeRetentionMetric`. ([tool-correctness](https://deepeval.com/docs/metrics-tool-correctness), [component-level](https://deepeval.com/docs/evaluation-component-level-llm-evals))

**Reproducibility / CI.** `deepeval test run` = pytest + flags (`-n` parallel, `-c` cache, `-r` repeat, `-d all|failing|passing`, `--official` baseline). Config objects `CacheConfig`/`AsyncConfig`/`DisplayConfig`/`ErrorConfig`. `@deepeval.log_hyperparameters`. CI via GH Actions; `assert_test` `AssertionError` → non-zero exit. ([unit-testing](https://deepeval.com/docs/evaluation-unit-testing-in-ci-cd))

**Report / terminal UX.** Per-case pass/fail with scores/thresholds/`reason`; `verbose_mode` shows judge reasoning; `DisplayConfig.file_type` writes HTML/MD reports; Confident AI web platform (`deepeval login`). (`deepeval view` command **unverified**.)

## 7. Inspect (UK AISI) — skim

Primary sources: [inspect.aisi.org.uk](https://inspect.aisi.org.uk/), `pip install inspect-ai`. Maintained by the UK AI Security Institute + Meridian Labs (exact current version not pinned — **check PyPI**).

**Distinctive model: `Task` = Dataset + Solver(s) + Scorer**, declared with `@task` for CLI discovery. ## Borrow / Adapt / Reject for a Ruby-first library

Tagged against the Ruby-first design principles: (1) Ruby-first not translated-Python; (2) DX over breadth; (3) core evaluates arbitrary Ruby callables, no Rails/RubyLLM dependency in core; (4) conventional objects/blocks/modules/matchers over elaborate abstractions; (5) simple simple, advanced possible; (6) deterministic first-class, LLM judges optional; (7) reproducible for local + CI.

### BORROW

- **The uniform scorer contract as a plain callable returning a small value object.** Braintrust's `Score{name, score, metadata}` and promptfoo's `{pass, score, reason}` converge; both deterministic and LLM judges use the *same* interface. **Idiomatic Ruby:** a scorer is any object responding to `#call(context) -> Result`, where `Result` is a small value object (`Struct`/`Data`) `Score.new(name:, value:, pass:, reason:)`; a block or lambda is a scorer too. Satisfies (3), (4), (5), (6).
- **Explicit result-kind value objects: `Assertion` / `Score` / `Label`.** Pydantic infers these from Python return type — Ruby can't, so make them **explicit tiny value objects** (or a matcher DSL). This is *better* than a magic-return convention and reads Ruby-native. `Data.define(:value, :reason)` for each. Satisfies (1), (4), (6).
- **The thin LLM-judge layer: rubric + choices + choice→score map + optional CoT, over the same scorer contract.** This recipe is now universal (Braintrust `LLMClassifier`, OpenAI modelgraded, Pydantic `LLMJudge`). **Idiomatic Ruby:** `LlmJudge.new(rubric:, choices: {pass: 1.0, fail: 0.0}, cot: true)` returning a `Score`; ships as an *optional* scorer, keeping the core deterministic (6). The judge model is injected, not baked in (3).
- **Grader model pluggable at clear precedence levels.** promptfoo's CLI < defaultTest < per-assertion precedence is a clean model. **Idiomatic Ruby:** a configurable default judge (`Evals.config.judge = …`) overridable per-suite and per-scorer via keyword. Satisfies (2), (6), (7).
- **Epochs + reducers with real variance reporting (Inspect).** This is the clearest differentiation opportunity — no Ruby lib does it and only Inspect does it well elsewhere. **Idiomatic Ruby:** `epochs: 5, reduce: :mean` (also `:median`, `:mode`, `:max`, `at_least(3)`, `pass_at(1)`), reported with a stderr column. Satisfies (5), (7) and is a genuine reason-to-exist.
- **Golden vs test-case split (DeepEval).** Decoupling inputs+expectations (the Golden) from the dynamically-generated `actual_output` lets the same dataset run across models/prompts. **Idiomatic Ruby:** a `Case`/`Golden` holding `input`/`expected`/`metadata`; the task callable fills in the output at run time. Satisfies (2), (5).
- **Dataset serialization to YAML/JSON/CSV with a code-first primary path.** Every code-first framework still serializes. **Idiomatic Ruby:** cases defined in Ruby (blocks) as the primary path, with `Dataset.load('cases.yml')`/`.save` for the data-driven path. Skip Pydantic's auto-JSON-Schema-generation initially (nice but heavy) — mark **Adapt**. Satisfies (1), (5).
- **CI ergonomics: meaningful exit codes + JUnit output.** promptfoo (exit 100, `junit.xml`) and DeepEval/Braintrust (non-zero on failure) make CI trivial. **Idiomatic Ruby:** non-zero exit on threshold failure, a `--format junit` reporter, and first-class RSpec/Minitest matchers (`expect(output).to satisfy_eval(...)`). Satisfies (2), (7).
- **The decompose→per-unit-verdict→aggregate judge idiom (Ragas/DeepEval).** Splitting an answer into atomic claims and scoring each reduces judge variance. **Idiomatic Ruby:** offer it as a composable judge builder, not a core requirement. Satisfies (5), (6) — **Adapt** (ship later).

### ADAPT

- **Weighted-average score composition + named metrics (promptfoo).** Weighted-average-of-assertions vs a test `threshold` is clean, but promptfoo's `derivedMetrics` math-expression strings are un-Ruby. **Adapt:** support per-scorer `weight:` and a test-level `threshold:`, but express derived metrics as **Ruby blocks over the named scores**, not string expressions. Satisfies (1), (4).
- **Agent/tool-call evaluation via a trace abstraction.** Pydantic (OTel span tree), DeepEval (`@observe` spans), promptfoo (OTLP) all need a trace representation. **Adapt:** define a minimal Ruby **trace/step value object** in core (tool name, args, output, timing) that arbitrary callables can emit — deliberately *not* coupled to RubyLLM/OTel in core (3). Deterministic `ToolCorrectness`/trajectory matchers over it (both DeepEval and Ragas keep these deterministic — good fit for principle 6). Satisfies (3), (6).
- **Response caching for reproducibility (promptfoo default-on disk cache; Ragas `DiskCacheBackend`).** Essential for cheap CI re-runs, but must be LLM/replay-aware (see the ecosystem scan's VCR caveat). **Adapt:** a purpose-built, opt-in cache keyed on request content, `--no-cache` to bypass when measuring real variance. Satisfies (7).
- **Two custom-judge archetypes: G-Eval (free-form) and DAG (deterministic decision tree).** Both are valuable but heavyweight. **Adapt:** ship a simple rubric/criteria judge first; offer a decision-tree/branching judge as an advanced, composable add-on (Ruby blocks as branch predicates). Satisfies (5).

### REJECT

- **Ragas' float-only, no-threshold, mean-only model.** Leaving pass/fail entirely to the caller conflicts with principle (6) (deterministic pass/fail first-class) and hurts DX (2). A Ruby lib should make thresholds/assertions first-class from day one. **Reject.**
- **OpenAI Evals' YAML-template-fused eval logic + registry.** Fusing evaluation logic into named YAML templates in a global registry is the opposite of Ruby-first callables (1, 3) and the project is quiescent anyway. Borrow only its JSONL-case-on-disk format, not its architecture. **Reject.**
- **Pydantic's return-type-driven result taxonomy (as a mechanism).** Elegant in a strongly-typed language, but relying on `bool` vs `int` vs `str` return types to mean assertion/score/label would be fragile and surprising in Ruby (`1`/`1.0`/`true` distinctions). Borrow the *concept* (three result kinds) but **reject** the type-magic mechanism in favor of explicit value objects. **Reject (mechanism).**
- **Config-as-data-first (promptfoo/OpenAI YAML-first) as the primary interface.** A single giant YAML config is portable but fights principle (1) and (4) — Ruby developers want blocks and objects. Support data-driven datasets, but the **primary** authoring path should be Ruby code, not YAML. **Reject (as primary path).**
- **Braintrust/DeepEval/Ragas platform-account coupling in core.** Cloud push/pull, hosted experiments, and account-gated reporting must never be in the core eval loop (3, 7 — CI must run offline). **Reject** any hosted dependency in core; make platform export an optional adapter.
